# 信号机制

信号是 Unix 异步通知机制——内核或其他进程向目标进程发送一个信号，目标进程处理（或被终止）。

## 信号本质

```
发送信号 → 内核设置目标进程的 pending 位图
         → 目标进程从内核态返回用户态时检查
         → 调用信号处理函数或执行默认动作
```

## 常见信号

| 信号 | 编号 | 默认动作 | 含义 |
|------|------|----------|------|
| SIGHUP | 1 | 终止 | 终端挂起 |
| SIGINT | 2 | 终止 | Ctrl+C |
| SIGKILL | 9 | 终止 | 强制杀死（不可捕获） |
| SIGSEGV | 11 | 终止+core | 段错误 |
| SIGPIPE | 13 | 终止 | 写无读端的管道 |
| SIGALRM | 14 | 终止 | 定时器到期 |
| SIGSTOP | 19 | 暂停 | 暂停进程（不可捕获） |
| SIGCHLD | 17 | 忽略 | 子进程状态变化 |

## xv6 的简化实现

xv6 只实现了 `kill` 系统调用——设置 `p->killed = 1`，进程在返回用户态时检查并退出。

```c
// trap.c
if(myproc() && myproc()->killed && (tf->cs&3) == DPL_USER)
  exit();

// syscall.c
int sys_kill(void) {
  argint(0, &pid);
  return proc_signal(pid, SIGKILL);  // 简化版
}
```

## 完整信号实现（Linux 风格）

### 信号处理结构

```c
struct sigaction {
  void (*handler)(int);  // 处理函数或 SIG_IGN/SIG_DFL
  sigset_t sa_mask;      // 处理期间屏蔽的信号
  int sa_flags;          // 标志
};
```

### 信号投递流程

```
1. 内核设置 pending |= (1 << signum)
2. 从内核态返回用户态前：
   a. 检查 pending & ~blocked
   b. 若有未屏蔽信号，准备信号帧
3. 信号帧布局：
   ┌─────────────────┐
   │ 原 trapframe    │
   │ 信号编号         │
   │ 返回地址 → sigreturn │
   └─────────────────┘
4. 跳转到 handler 执行
5. handler 返回 → 调用 sigreturn
6. sigreturn 恢复原 trapframe → 回到被打断的地方
```

### sigreturn

```c
// 从信号帧恢复现场，继续执行
int sys_sigreturn(void) {
  struct trapframe *tf = myproc()->tf;
  // 从用户栈恢复原 trapframe
  // 恢复信号掩码
  return tf->eax;  // 返回原系统调用的返回值
}
```

## 信号 vs 异常

| | 信号 | 异常/中断 |
|---|------|----------|
| 来源 | 内核/其他进程 | 硬件/软件 |
| 投递 | 异步（延迟到返回用户态） | 同步（立即） |
| 处理 | 用户态 handler | 内核态 handler |
| 可否屏蔽 | 可以（sigprocmask） | 不可以 |

## 关键要点

> 信号是 Unix IPC 中最古老也最混乱的机制——异步、可丢失、处理函数有各种边界情况。xv6 只实现了最简化的 kill（标记 killed），但完整信号机制的核心是**从内核态返回用户态时的检查点**——这是投递信号的唯一时机。理解信号帧和 sigreturn 是理解用户态异常处理的关键。
