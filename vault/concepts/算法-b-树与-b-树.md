# B 树与 B+ 树

B 树是多路平衡搜索树，每个节点存多个 key，专为减少磁盘 I/O 设计。B+ 树是其变体，是现代数据库索引的核心。

## B 树

### 定义（m 阶 B 树）

1. 每个节点最多 m 个 key，m+1 个子节点
2. 除根和叶外，每个节点至少 ⌈m/2⌉ 个 key
3. 根至少 1 个 key（除非空树）
4. 所有叶子在同一层
5. 节点内 key 有序排列

```
      [10 | 20]              ← 2 个 key, 3 个子
     /    |    \
  [3|5] [12|15] [25|30|35]  ← 叶子节点
```

### 查找

```python
def btree_search(node, key):
    i = 0
    while i < len(node.keys) and key > node.keys[i]:
        i += 1
    if i < len(node.keys) and key == node.keys[i]:
        return (node, i)
    if node.is_leaf:
        return None
    return btree_search(node.children[i], key)
```

**时间**：O(log_m(n))，m 通常很大（如 512），树高极低（3~4 层即可存百万数据）。

### 插入（溢出时分裂）

```
节点满 (m 个 key) → 取中位数提升到父节点 → 分裂为两个节点
```

### 复杂度

| 操作 | 时间 | 说明 |
|------|------|------|
| 查找 | O(log_m n) | m 大则树高很小 |
| 插入 | O(log_m n) | 分裂向上传播 |
| 删除 | O(log_m n) | 合并/借用 |

## B+ 树

### 与 B 树的区别

| 特性 | B 树 | B+ 树 |
|------|------|-------|
| 数据位置 | 所有节点 | **仅叶子节点** |
| 叶子链接 | 无 | 叶子间链表相连 |
| 范围查询 | 需中序遍历 | ✅ 叶子链表顺序扫描 |
| 内部节点 | 存 key + data | 只存 key（不存 data） |

### B+ 树结构

```
内部节点（只有 key，用于路由）：
      [10 | 20]
     /    |    \
叶子节点（存 key + data，链表连接）：
  [3→5] → [10→12→15] → [20→25→30→35]
  ↑ 链表串联，便于范围查询
```

**内部节点不存数据** → 每个节点可容纳更多 key → 树更矮 → 更少磁盘 I/O。

## 数据库中的应用

### MySQL InnoDB

```
InnoDB 使用 B+ 树作为索引结构：

主键索引（Clustered Index）：
- 叶子节点存完整行数据
- 一棵 B+ 树 = 整张表

二级索引（Secondary Index）：
- 叶子节点存主键值
- 查到主键后还需回表查完整数据

页大小：16KB
扇出：通常 100~200（取决于 key 大小）
三层 B+ 树可索引约 100^3 = 100 万条记录
```

### 为什么数据库选 B+ 树不选哈希

| 对比 | B+ 树 | 哈希索引 |
|------|-------|----------|
| 范围查询 | ✅ 叶子链表 | ❌ 不支持 |
| 排序 | ✅ | ❌ |
| 前缀匹配 | ✅ | ❌ |
| 精确查询 | O(log n) | ✅ O(1) |

## 关键要点

> [!important] B+ 树 = 数据库索引的灵魂
> MySQL InnoDB、PostgreSQL 都基于 B+ 树索引

> [!important] 树高极低
> 3~4 层的 B+ 树可索引上亿条记录，每次查询只需 3~4 次磁盘 I/O

> [!important] 叶子链表是关键设计
> 范围查询只需定位到起始叶子，然后沿链表顺序扫描
