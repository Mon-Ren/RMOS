# Chain of Responsibility 责任链模式

将请求沿着处理者链传递，每个处理者决定处理请求或转发给下一个。

## 意图与场景

- **核心意图**：避免请求发送者与接收者之间的耦合，让多个对象都有机会处理请求
- **典型场景**：
  - 日志系统（按级别过滤：DEBUG → INFO → WARN → ERROR）
  - HTTP 中间件链（认证 → 鉴权 → 限流 → 业务处理）
  - GUI 事件冒泡（子组件 → 父组件 → 窗口）
  - 异常处理链
  - 审批流程（主管 → 经理 → 总监）

## C++ 实现

### 经典实现：抽象处理者 + next 指针

```cpp
#include <memory>
#include <string>
#include <iostream>

// 请求
struct HttpRequest {
    std::string path;
    std::string token;
    std::string body;
};

// 抽象处理者
class Middleware {
protected:
    std::unique_ptr<Middleware> next_;
public:
    virtual ~Middleware() = default;

    // 链式设置下一个处理者
    std::unique_ptr<Middleware> setNext(std::unique_ptr<Middleware> handler) {
        next_ = std::move(handler);
        return next_ ? nullptr : nullptr; // 返回自身以支持链式调用
    }

    // 模板方法：尝试处理或转发
    virtual void handle(const HttpRequest& req) {
        if (next_) {
            next_->handle(req);
        }
    }
};

// 具体处理者：认证中间件
class AuthMiddleware : public Middleware {
public:
    void handle(const HttpRequest& req) override {
        if (req.token.empty()) {
            std::cout << "[Auth] 拒绝：缺少 token\n";
            return; // 链中断
        }
        std::cout << "[Auth] token 验证通过\n";
        if (next_) next_->handle(req); // 转发
    }
};

// 具体处理者：限流中间件
class RateLimitMiddleware : public Middleware {
    int requestCount_ = 0;
    int limit_;
public:
    explicit RateLimitMiddleware(int limit) : limit_(limit) {}

    void handle(const HttpRequest& req) override {
        if (++requestCount_ > limit_) {
            std::cout << "[RateLimit] 拒绝：超过限流阈值\n";
            return;
        }
        std::cout << "[RateLimit] 通过 (" << requestCount_ << "/" << limit_ << ")\n";
        if (next_) next_->handle(req);
    }
};

// 具体处理者：业务处理
class BusinessHandler : public Middleware {
public:
    void handle(const HttpRequest& req) override {
        std::cout << "[Business] 处理请求: " << req.path << "\n";
        // 链末端，不转发
    }
};
```

### 现代风格：基于 `std::function` 的轻量链

```cpp
#include <functional>
#include <vector>
#include <optional>

using Handler = std::function<bool(const HttpRequest&)>;

class MiddlewareChain {
    std::vector<Handler> handlers_;
public:
    void add(Handler h) { handlers_.push_back(std::move(h)); }

    // 依次执行，任一返回 true 则中断
    bool process(const HttpRequest& req) const {
        for (const auto& h : handlers_) {
            if (h(req)) return true; // 已处理，中断链
        }
        return false; // 无人处理
    }
};

// 使用
// chain.add([](const auto& req) { /* auth check */ return false; });
// chain.add([](const auto& req) { /* business logic */ return true; });
// chain.process(req);
```

## 优缺点

| 优点 | 缺点 |
|------|------|
| 降低耦合：发送者无需知道具体接收者 | 请求可能无人处理（需兜底策略） |
| 符合开闭原则：新增处理者不动已有代码 | 链过长时调试困难，性能下降 |
| 灵活组合处理链 | 难以保证请求一定被处理 |
| 单一职责：每个处理者只关心自己的逻辑 | 链的顺序可能影响行为 |

## 关键要点

> **核心思想**：将请求的发送者和接收者解耦，让多个处理者都有机会处理请求，形成隐式的链式调用。

> **现代 C++ 建议**：对于简单场景，`std::function` + `vector` 比继承链更轻量；对于复杂中间件（需要 pre/post 处理），经典 OOP 实现更合适。

> **常见变体**：
> - **纯责任链**：每个请求最多被一个处理者处理
> - **功能链**：每个处理者都对请求做加工（类似 [[设计模式-装饰器模式|Decorator]]）

> **关键决策**：处理者是**独占**还是**共享**？是否需要保证链的完整性（比如必须有人处理）？

## 相关模式

- [[设计模式-装饰器模式|Decorator 模式]]：结构类似，但 CoR 的处理者可以选择不传递，Decorator 则是每层都执行
- [[设计模式-命令模式|Command 模式]]：可以将命令对象放入责任链中处理
- [[设计模式-中介者模式|Mediator 模式]]：CoR 是链式分发，Mediator 是集中式协调
