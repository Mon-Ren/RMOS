# 创建型模式对比

五种创建型模式解决的核心问题都是"如何创建对象"，但各自的侧重点截然不同。

## 一览表

| 模式 | 核心意图 | 关键词 | 典型场景 | 复杂度 |
|------|----------|--------|----------|--------|
| [[设计模式-单例模式\|Singleton]] | 全局唯一实例 | 唯一、全局 | 日志、配置、连接池 | ⭐ |
| [[设计模式-工厂方法模式\|Factory Method]] | 延迟到子类决定实例化哪个类 | 解耦、扩展 | 日志器、解析器、插件 | ⭐⭐ |
| [[设计模式-抽象工厂模式\|Abstract Factory]] | 创建一族相关对象 | 产品族、一致 | 跨平台 UI、多数据库 | ⭐⭐⭐ |
| [[设计模式-建造者模式\|Builder]] | 分步构建复杂对象 | 链式、可选参数 | SQL/HTTP 构建、复杂配置 | ⭐⭐ |
| [[设计模式-原型模式\|Prototype]] | 通过克隆创建对象 | 复制、快照 | 游戏实体、撤销机制 | ⭐⭐ |

## 什么时候用哪个

### 简单创建 → 工厂方法
> "我不知道要创建什么类型，让工厂决定。"

适用于：对象类型在运行时确定，但创建逻辑简单的场景。

```cpp
auto logger = LoggerFactory::create(config.type());
```

### 复杂参数 → Builder
> "我要创建的对象有太多可选参数和构造步骤。"

适用于：构造函数参数超过 4-5 个，或构造有顺序依赖。

```cpp
auto request = RequestBuilder{}
    .url("https://api.example.com")
    .method("POST")
    .header("Content-Type", "application/json")
    .body(data)
    .build();
```

### 一族对象 → 抽象工厂
> "我需要一组相关的对象，它们必须协同工作。"

适用于：多平台/多数据库/多主题的场景，需要保证产品族一致性。

```cpp
std::unique_ptr<UIFactory> factory = get_factory(os_type);
auto button = factory->create_button();  // 与当前 OS 匹配
auto menu = factory->create_menu();
```

### 克隆已有对象 → Prototype
> "创建这个对象代价太高，我更想复制一个现成的。"

适用于：对象初始化昂贵，或需要大量相似对象。

```cpp
auto enemy = prototype_registry.create("goblin");
// 基于模板创建，无需重新加载纹理/配置
```

### 全局唯一 → Singleton
> "整个系统只需要一个实例。"

适用于：日志、配置、硬件接口等天然唯一的资源。

```cpp
Logger::instance().log("ready");
```

> ⚠️ **Singleton 是最后手段**。优先考虑依赖注入——它让依赖关系显式化，也更容易测试。

## 决策流程图

```
需要创建对象
    │
    ├─ 全局只需一个？ ──→ Singleton（慎重使用）
    │
    ├─ 已有对象可以克隆？ ──→ Prototype
    │
    ├─ 需要一族相关对象？ ──→ Abstract Factory
    │
    ├─ 构造参数多/步骤复杂？ ──→ Builder
    │
    └─ 其他 → Factory Method（或直接 new）
```

## C++ 特有考量

| 特性 | 对创建型模式的影响 |
|------|-------------------|
| `std::unique_ptr` | 工厂/原型的返回类型，天然表达所有权转移 |
| `std::make_unique` | 替代 `new`，异常安全 |
| 模板/CRTP | 编译时多态替代虚函数工厂，零开销 |
| 移动语义 | Builder 链式调用的基础 |
| RAII | Singleton 的 Meyers 实现依赖静态析构 |

## 关键要点

> **大多数项目只需要 Factory Method + Builder**。抽象工厂和原型用于特定场景，Singleton 能不用就不用。

> **创建型模式的本质是将 `new` 操作抽象化**。无论是工厂、Builder 还是原型，目的都是让调用者不直接依赖具体类的构造函数。

> **现代 C++ 的最佳实践**：用 `unique_ptr` 管理生命周期，用 `make_unique` 创建对象，用依赖注入替代 Singleton。
