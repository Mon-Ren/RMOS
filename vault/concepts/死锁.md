# 死锁

死锁是多个进程/线程互相等待对方持有的资源，导致全体阻塞、无法推进的状态。

## 四个必要条件（Coffman 条件）

全部满足才会死锁：

1. **互斥（Mutual Exclusion）** — 资源一次只能被一个进程持有
2. **持有并等待（Hold and Wait）** — 持有资源的同时等待更多资源
3. **不可抢占（No Preemption）** — 资源只能由持有者主动释放
4. **循环等待（Circular Wait）** — 进程链 P1→P2→...→Pn→P1 形成等待环

## 经典示例

```
线程 A:                    线程 B:
lock(m1);                  lock(m2);
lock(m2);  ← 阻塞          lock(m1); ← 阻塞
  ↓                          ↓
等 B 释放 m2               等 A 释放 m1
  └────── 死锁 ──────────────┘
```

## 处理策略

### 1. 鸵鸟策略（Ignore）

大多数 OS 选择：不检测、不预防，靠程序员避免。xv6 就是这样——因为锁的使用模式简单（内核开发者设计好的）。

### 2. 预防（Prevention）

破坏四个条件之一：

| 破坏的条件 | 方法 |
|-----------|------|
| 互斥 | 用无锁数据结构（CAS、RCU） |
| 持有并等待 | 一次性申请所有资源 |
| 不可抢占 | 申请不到就释放已有资源 |
| 循环等待 | **全局锁序**——最常用 |

### 3. 避免（Avoidance）— 银行家算法

运行时动态判断：分配资源后系统是否安全？

```
安全 = 存在一种执行顺序，所有进程都能完成
```

需要预先知道最大资源需求，实际中很少用。

### 4. 检测与恢复

- 构建**资源分配图**，检测环路
- 发现死锁后：回滚进程、杀死某个进程、抢占资源

## 实践中的做法

**Linux 内核的策略：**
- `lockdep` — 编译时/运行时检测锁序违规
- 自定义锁序规则：按地址大小排序、按锁类型分层
- `mutex_trylock()` — 非阻塞尝试，失败就回退

**xv6 的做法：**
- 全局锁序：`ptable.lock` > `sleep lock` > `inode lock`
- 持有 ptable.lock 时不能 acquire 其他锁
- 简单但有效

## 关键要点

> 实际工程中，**约定全局锁序**是防止死锁最实用的方法。检测和恢复机制用于兜底，但最好永远不触发。
