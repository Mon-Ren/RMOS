# Decorator 装饰器模式

动态地为对象添加额外职责，而不改变其接口。装饰器是被装饰对象的透明包装，客户端无需知道额外功能的存在。

## 意图与场景

继承也能添加功能，但会导致类爆炸且是静态的（编译时确定）。Decorator 在运行时通过组合灵活地叠加功能。

**典型场景：**
- I/O 流：`std::ostream` 的 decorator 设计（`cout` → `ofstream` → `ostringstream`）
- 中间件链：HTTP 过滤器、日志装饰
- 数据压缩/加密层叠加

**与继承的区别：**

| | 继承 | Decorator |
|---|---|---|
| 时间 | 编译时确定 | 运行时组合 |
| 灵活性 | 固定层级 | 任意叠加顺序 |
| 类数量 | 组合爆炸 | 线性增长 |

## C++ 实现

### 经典装饰器

```cpp
#include <iostream>
#include <memory>
#include <string>

// --- Component 接口 ---
class DataSource {
public:
    virtual ~DataSource() = default;
    virtual void write(const std::string& data) = 0;
    virtual std::string read() = 0;
};

// --- Concrete Component ---
class FileDataSource : public DataSource {
    std::string filename_;
    std::string data_;
public:
    explicit FileDataSource(std::string filename)
        : filename_(std::move(filename)) {}

    void write(const std::string& data) override {
        data_ = data;
        std::cout << "Written to " << filename_ << "\n";
    }

    std::string read() override {
        return data_;
    }
};

// --- Base Decorator ---
class DataSourceDecorator : public DataSource {
protected:
    std::unique_ptr<DataSource> wrappee_;
public:
    explicit DataSourceDecorator(std::unique_ptr<DataSource> source)
        : wrappee_(std::move(source)) {}

    void write(const std::string& data) override {
        wrappee_->write(data);
    }

    std::string read() override {
        return wrappee_->read();
    }
};

// --- Concrete Decorator: 加密 ---
class EncryptionDecorator : public DataSourceDecorator {
public:
    using DataSourceDecorator::DataSourceDecorator;

    void write(const std::string& data) override {
        auto encrypted = "[ENC]" + data + "[/ENC]";
        wrappee_->write(encrypted);
    }

    std::string read() override {
        auto raw = wrappee_->read();
        // 简化解密
        return raw.substr(5, raw.size() - 11);
    }
};

// --- Concrete Decorator: 压缩 ---
class CompressionDecorator : public DataSourceDecorator {
public:
    using DataSourceDecorator::DataSourceDecorator;

    void write(const std::string& data) override {
        auto compressed = "[ZIP]" + data + "[/ZIP]";
        wrappee_->write(compressed);
    }

    std::string read() override {
        auto raw = wrappee_->read();
        return raw.substr(4, raw.size() - 9);
    }
};

// --- 使用：运行时灵活叠加 ---
int main() {
    // 基础：普通文件
    auto source = std::make_unique<FileDataSource>("data.txt");

    // 叠加加密
    auto encrypted = std::make_unique<EncryptionDecorator>(std::move(source));

    // 再叠加压缩
    auto compressed = std::make_unique<CompressionDecorator>(std::move(encrypted));

    compressed->write("Hello, World!");
    // 写入链：压缩 → 加密 → 文件

    auto result = compressed->read();
    std::cout << "Read: " << result << "\n";  // Hello, World!
}
```

### std::ostream 的 Decorator 设计

标准库的 I/O 流是 Decorator 模式的经典案例：

```cpp
#include <iostream>
#include <fstream>
#include <sstream>

int main() {
    // 基础：标准输出
    std::ostream& os = std::cout;

    // 文件流是"装饰"——用文件缓冲替代标准输出缓冲
    std::ofstream file("output.txt");
    file << "写入文件\n";

    // 字符串流也是装饰
    std::ostringstream oss;
    oss << "写入字符串: " << 42;
    auto result = oss.str();  // "写入字符串: 42"

    // 输出重定向也是装饰思想
    auto* old_buf = std::cout.rdbuf(file.rdbuf());
    std::cout << "这段输出到文件\n";
    std::cout.rdbuf(old_buf);  // 恢复
}
```

### 编译时装饰（CRTP / 模板）

现代 C++ 也可以在编译时完成装饰，避免虚函数开销：

```cpp
#include <iostream>

// CRTP 编译时装饰
template <typename T>
class TimestampDecorator {
    T inner_;
public:
    template <typename... Args>
    explicit TimestampDecorator(Args&&... args)
        : inner_(std::forward<Args>(args)...) {}

    void log(const std::string& msg) {
        std::cout << "[2026-04-04 22:00] ";
        inner_.log(msg);
    }
};

class SimpleLogger {
public:
    void log(const std::string& msg) {
        std::cout << msg << "\n";
    }
};

int main() {
    TimestampDecorator<SimpleLogger> logger;
    logger.log("系统启动");
    // 输出: [2026-04-04 22:00] 系统启动
}
```

## 优缺点

**优点：**
- 运行时灵活组合，避免继承导致的类爆炸
- 单一职责：每个装饰器只关注一个增强功能
- 符合开闭原则：新增装饰器不影响已有代码

**缺点：**
- 大量小对象，调试时难以追踪完整调用链
- 装饰顺序敏感，不同顺序可能产生不同结果
- 移除某个装饰器较困难

## 关键要点

> - Decorator 是「透明包装」——接口不变，行为增强
> - C++ 标准库的 I/O 流体系是 Decorator 模式的经典应用
> - 与继承的静态扩展不同，Decorator 支持运行时动态叠加
> - 现代 C++ 中，模板/CRTP 可实现零开销的编译时装饰
> - 与 [[设计模式-代理模式|Proxy]] 的区别：Proxy 控制访问；Decorator 增强功能
