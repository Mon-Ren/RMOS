# 原子指令与内存屏障

多核并发的基础——没有原子操作就没有锁，没有内存屏障就没有正确的并发。

## x86 原子指令

### xchg — 原子交换

```c
// 原子地将 src 写入 *dst，返回旧值
// 整个操作对其他 CPU 不可见中间状态
uint xchg(uint *dst, uint src);
```

xv6 自旋锁的核心：
```c
while(xchg(&lk->locked, 1) != 0)
  ;  // 忙等直到获取锁
```

### 其他原子指令

| 指令 | 操作 | 用途 |
|------|------|------|
| `xchg` | 交换 | 自旋锁 |
| `cmpxchg` | 比较并交换 (CAS) | 无锁数据结构 |
| `xadd` | 原子加 | 计数器 |
| `lock add` | 带锁前缀的加 | 通用原子操作 |

**`lock` 前缀**：使后续指令对总线/缓存行加锁，在多核环境下保证原子性。

## 内存屏障

CPU 和编译器都会**重排指令**以优化性能。在并发场景下，重排可能导致错误。

### 编译器屏障

```c
__sync_synchronize();  // GCC 内建
```

阻止编译器将此屏障前后的指令重排。

### 硬件屏障（x86）

x86 有强内存模型（TSO），大多数情况不需要显式屏障。但以下场景仍需要：

| 指令 | 作用 |
|------|------|
| `mfence` | 全屏障：之前的所有读写完成后才能继续 |
| `lfence` | 读屏障 |
| `sfence` | 写屏障 |

## xv6 中的使用

```c
void acquire(struct spinlock *lk) {
  while(xchg(&lk->locked, 1) != 0)  // 原子获取
    ;
  __sync_synchronize();  // 内存屏障：确保临界区操作在锁获取之后
}

void release(struct spinlock *lk) {
  __sync_synchronize();  // 内存屏障：确保临界区操作在锁释放之前完成
  asm volatile("movl $0, %0" : "+m" (lk->locked));  // 释放锁
}
```

### 为什么需要屏障？

```
没有屏障时可能的执行顺序（CPU 重排）：

CPU0:                  CPU1:
  data = 42;            while(flag == 0);
  flag = 1;             print(data);  // 可能读到旧值！

CPU 可能将 flag=1 重排到 data=42 之前。
屏障确保 data=42 对其他 CPU 可见后，flag=1 才可见。
```

## 关键要点

> x86 原子指令是并发编程的基石——`xchg` 保证读-改-写的原子性，`lock` 前缀扩展到更复杂操作。内存屏障解决指令重排问题：x86 虽然是 TSO（Total Store Order）强模型，但获取锁时仍需屏障确保临界区内存操作的可见性。xv6 的 `__sync_synchronize()` 就是编译器+硬件的双重屏障。
