# 工厂方法模式 (Factory Method)

定义一个创建对象的接口，让子类决定实例化哪个类——将实例化推迟到子类。

## 意图与场景

**核心问题**：调用者需要创建对象，但不应依赖具体类。新增产品类型时，不应该修改已有代码（OCP）。

**典型场景**：
- 日志系统（文件日志、控制台日志、远程日志）
- GUI 框架（不同平台的按钮/窗口）
- 解析器（JSON、XML、YAML）
- 游戏中的敌人/道具创建

### 简单工厂 vs 工厂方法 vs 抽象工厂

| | 简单工厂 | 工厂方法 | 抽象工厂 |
|--|----------|----------|----------|
| 创建方式 | 静态方法 + switch | 虚函数，子类重写 | 多个工厂方法组合 |
| 扩展方式 | 修改 switch（违反 OCP） | 新增子类 | 新增工厂子类 |
| 产品族 | 单一产品 | 单一产品 | 一族产品 |
| 复杂度 | 低 | 中 | 高 |

## C++ 实现

### 基础实现：虚函数 + 工厂方法

```cpp
#include <memory>
#include <string>
#include <string_view>
#include <iostream>
#include <fstream>

// 产品接口
class Logger {
public:
    virtual ~Logger() = default;
    virtual void log(std::string_view msg) = 0;
};

// 具体产品
class ConsoleLogger : public Logger {
public:
    void log(std::string_view msg) override {
        std::cout << "[Console] " << msg << "\n";
    }
};

class FileLogger : public Logger {
    std::ofstream file_;
public:
    explicit FileLogger(std::string_view path) : file_(path.data(), std::ios::app) {}
    void log(std::string_view msg) override {
        file_ << "[File] " << msg << "\n";
    }
};

// 工厂方法
class LoggerFactory {
public:
    virtual ~LoggerFactory() = default;
    virtual std::unique_ptr<Logger> create_logger() = 0;
};

class ConsoleLoggerFactory : public LoggerFactory {
public:
    std::unique_ptr<Logger> create_logger() override {
        return std::make_unique<ConsoleLogger>();
    }
};

class FileLoggerFactory : public LoggerFactory {
    std::string path_;
public:
    explicit FileLoggerFactory(std::string_view path) : path_(path) {}
    std::unique_ptr<Logger> create_logger() override {
        return std::make_unique<FileLogger>(path_);
    }
};

// 客户端代码 — 只依赖抽象
void run_app(LoggerFactory& factory) {
    auto logger = factory.create_logger();
    logger->log("Hello from factory method!");
}
```

### 注册表模式（避免硬编码子类）

```cpp
#include <unordered_map>
#include <functional>
#include <stdexcept>

class LoggerRegistry {
public:
    using Creator = std::function<std::unique_ptr<Logger>()>;

    static LoggerRegistry& instance() {
        static LoggerRegistry reg;
        return reg;
    }

    void register_type(std::string_view name, Creator creator) {
        creators_[std::string(name)] = std::move(creator);
    }

    std::unique_ptr<Logger> create(std::string_view name) const {
        auto it = creators_.find(std::string(name));
        if (it == creators_.end())
            throw std::invalid_argument("Unknown logger type: " + std::string(name));
        return it->second();
    }

private:
    LoggerRegistry() = default;
    std::unordered_map<std::string, Creator> creators_;
};

// 注册（程序启动时）
```

### 模板工厂（减少样板代码）

```cpp
template <typename Product, typename... Args>
class GenericFactory {
public:
    static std::unique_ptr<Product> create(Args... args) {
        return std::make_unique<Product>(std::forward<Args>(args)...);
    }
};

// 使用
// auto logger = GenericFactory<ConsoleLogger>::create();
```

## 优缺点

### ✅ 优点
- 符合开闭原则：新增产品只需新增工厂子类
- 调用者与具体产品解耦
- 单一职责：创建逻辑集中在一个类中

### ❌ 缺点
- 类数量增加（每个产品一对 Factory + Product）
- 引入抽象层，增加理解成本
- 简单场景下过度设计

## 关键要点

> **工厂方法的核心是"延迟决策"**：不在编译时绑定具体类型，而是在运行时由工厂决定创建什么。这正是 DIP（依赖倒置原则）的体现。

> **简单工厂 → 工厂方法 → 抽象工厂** 是递进关系。大多数项目用简单工厂就够了；需要开闭原则时升级到工厂方法；需要产品族时再用抽象工厂。

> **注册表模式**是工厂方法的现代变体：用 `string → lambda` 的 map 替代虚函数继承树，更灵活，且支持运行时注册。
