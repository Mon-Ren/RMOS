# 单例模式 (Singleton)

确保一个类只有一个实例，并提供一个全局访问点。

## 意图与场景

**核心问题**：某些对象全局只需要一份——配置管理器、日志系统、线程池、数据库连接池。

**典型场景**：
- 日志系统（全局唯一的 Logger）
- 配置管理（读取一次，全局共享）
- 硬件接口（打印机驱动、显卡上下文）
- 缓存/注册表

## C++ 实现

### Meyers' Singleton（推荐）

C++11 保证静态局部变量的初始化是线程安全的，这是最简洁的单例实现：

```cpp
class Logger {
public:
    static Logger& instance() {
        static Logger instance;  // C++11 保证线程安全
        return instance;
    }

    // 删除拷贝和移动
    Logger(const Logger&) = delete;
    Logger& operator=(const Logger&) = delete;
    Logger(Logger&&) = delete;
    Logger& operator=(Logger&&) = delete;

    void log(std::string_view msg) {
        // ...
    }

private:
    Logger() = default;
};

// 使用
Logger::instance().log("Application started");
```

### `std::call_once` 替代方案

适用于需要延迟初始化且有复杂构造逻辑的场景：

```cpp
class DatabasePool {
public:
    static DatabasePool& instance() {
        static std::once_flag flag;
        std::call_once(flag, []() {
            instance_ = std::unique_ptr<DatabasePool>(new DatabasePool());
        });
        return *instance_;
    }

private:
    DatabasePool() { /* 复杂初始化 */ }
    static inline std::unique_ptr<DatabasePool> instance_;
};
```

### 双重检查锁定（历史遗留，不推荐）

C++11 之前的做法，在内存模型未标准化时容易出 bug：

```cpp
// ⚠️ 不推荐 — 仅作教学用途
class Singleton {
public:
    static Singleton* instance() {
        if (instance_ == nullptr) {           // 第一次检查（无锁）
            std::lock_guard lock(mutex_);
            if (instance_ == nullptr) {       // 第二次检查（有锁）
                instance_ = new Singleton();
            }
        }
        return instance_;
    }
private:
    static inline std::atomic<Singleton*> instance_{nullptr};
    static inline std::mutex mutex_;
    Singleton() = default;
};
```

> C++11 前，指令重排序可能导致返回未完全构造的对象。`std::atomic` 的 acquire-release 语义可修复，但复杂度远高于 Meyers' Singleton。

## 优缺点

### ✅ 优点
- 全局唯一访问点，易于协调
- 延迟初始化（首次使用时才创建）
- 避免全局变量的初始化顺序问题

### ❌ 缺点
- **全局状态**：隐藏了依赖关系，违反依赖注入原则
- **测试困难**：难以 mock，测试间可能互相污染
- **违反 SRP**：既管理业务逻辑，又管理自身生命周期
- **多线程销毁顺序**：程序退出时析构顺序不确定（跨编译单元的单例互相引用时）

## 关键要点

> **Meyers' Singleton 是 C++11+ 的标准做法**：一行 `static` 局部变量，线程安全、简洁、零额外开销。不需要 `call_once`，不需要双重检查锁定。

> **单例的本质问题是可测试性**。现代 C++ 更推荐通过依赖注入传递对象，而非使用全局单例。如果发现自己频繁使用单例，考虑是否该用 `main()` 中创建对象并显式传递。

> **替代方案**：依赖注入容器、`std::shared_ptr` 在 `main()` 中创建并传递、或仅在受限的工厂函数内控制实例数量。
