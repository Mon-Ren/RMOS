# x86 系统调用演进

## 三代系统调用机制

### 第一代：int 0x80（xv6 使用）

```asm
mov eax, SYS_write
mov ebx, fd
mov ecx, buf
mov edx, len
int 0x80
```

- 通过软件中断陷入内核
- CPU：查 IDT → 压栈 → 跳转处理函数
- **开销：~1000+ 周期**（IDT 查找 + 两次特权级切换 + 栈操作）

### 第二代：sysenter/sysexit（Intel, P6+）

```asm
; 用户态
mov edx, esp        ; 保存用户栈
mov ecx, eip        ; 保存返回地址
sysenter             ; 快速进入内核

; 内核态入口（MSR 预设）
; sysexit 返回用户态
```

- 专用 MSR 寄存器预设内核入口和栈
- 不查 IDT、不压完整栈
- **开销：~300 周期**

### 第三代：syscall/sysret（AMD, K6-2+）

```asm
; 用户态
mov eax, SYS_write
mov rdi, fd         ; 参数走寄存器（64位）
mov rsi, buf
mov rdx, len
syscall
```

- 更简单：RCX 保存 RIP，R11 保存 RFLAGS
- 内核入口在 LSTAR MSR 中预设
- **开销：~200 周期**
- x86-64 Linux 的默认选择

## 性能对比

```
int 0x80:     ~1000 周期  ← xv6 使用
sysenter:     ~300 周期
syscall:      ~200 周期   ← Linux x86-64
vDSO:         ~0 周期     ← 不陷入内核（gettimeofday 等）
```

## vDSO（virtual Dynamic Shared Object）

某些系统调用（如 `gettimeofday`、`clock_gettime`）不需要真正陷入内核——内核将结果映射到用户空间只读页。

```c
// 用户态直接读取，零开销
struct timeval *tv = vdso_gettimeofday_addr;
return tv->tv_sec;
```

## x86-64 参数传递

64 位下系统调用参数走寄存器（类似 fastcall）：

| 参数 | 寄存器 |
|------|--------|
| syscall 号 | RAX |
| arg1 | RDI |
| arg2 | RSI |
| arg3 | RDX |
| arg4 | R10 |
| arg5 | R8 |
| arg6 | R9 |
| 返回值 | RAX |

对比 xv6-32：所有参数走栈（`tf->esp + 偏移`）。

## 为什么 xv6 用 int 0x80？

- 简单——硬件直接支持，不需要设置 MSR
- 教学——完整展示了系统调用的硬件机制
- 兼容——x86-32 时代的标准做法

## 关键要点

> 系统调用的演进反映了**内核/用户态切换开销**的优化历史。int 0x80 通过 IDT 跳转，路径最长；sysenter/syscall 用预设 MSR 绕过 IDT，大幅缩短路径；vDSO 更激进——某些调用根本不进入内核。参数传递也从栈演进到寄存器，减少内存访问。xv6 用 int 0x80 是教学选择，不是性能选择。
