# 抽象工厂模式 (Abstract Factory)

提供一个接口，用于创建**一族相关或相互依赖的对象**，而无需指定它们的具体类。

## 意图与场景

**核心问题**：系统需要创建多个相关的产品对象，且这些对象必须属于同一产品族。

**典型场景**：
- 跨平台 UI（Win/Mac/Linux 各自的 Button、Menu、Dialog）
- 数据库访问层（MySQL/PostgreSQL 各自的 Connection、Command、Reader）
- 游戏中的主题系统（不同主题的敌人、武器、地形）

**与工厂方法的区别**：工厂方法创建**一个**产品，抽象工厂创建**一族**产品。

## C++ 实现

### 跨平台 UI 工厂示例

```cpp
#include <memory>
#include <iostream>
#include <string_view>

// ===== 抽象产品 =====
class Button {
public:
    virtual ~Button() = default;
    virtual void render() const = 0;
};

class Menu {
public:
    virtual ~Menu() = default;
    virtual void render() const = 0;
};

// ===== 具体产品：Windows =====
class WinButton : public Button {
public:
    void render() const override {
        std::cout << "[Win] Button rendered\n";
    }
};

class WinMenu : public Menu {
public:
    void render() const override {
        std::cout << "[Win] Menu rendered\n";
    }
};

// ===== 具体产品：Mac =====
class MacButton : public Button {
public:
    void render() const override {
        std::cout << "[Mac] Button rendered\n";
    }
};

class MacMenu : public Menu {
public:
    void render() const override {
        std::cout << "[Mac] Menu rendered\n";
    }
};

// ===== 抽象工厂 =====
class UIFactory {
public:
    virtual ~UIFactory() = default;
    virtual std::unique_ptr<Button> create_button() const = 0;
    virtual std::unique_ptr<Menu> create_menu() const = 0;
};

// ===== 具体工厂 =====
class WinFactory : public UIFactory {
public:
    std::unique_ptr<Button> create_button() const override {
        return std::make_unique<WinButton>();
    }
    std::unique_ptr<Menu> create_menu() const override {
        return std::make_unique<WinMenu>();
    }
};

class MacFactory : public UIFactory {
public:
    std::unique_ptr<Button> create_button() const override {
        return std::make_unique<MacButton>();
    }
    std::unique_ptr<Menu> create_menu() const override {
        return std::make_unique<MacMenu>();
    }
};

// ===== 客户端代码 =====
void build_ui(const UIFactory& factory) {
    auto button = factory.create_button();
    auto menu = factory.create_menu();
    button->render();
    menu->render();
}

// 使用
// WinFactory win;
// build_ui(win);  // 全部是 Win 风格组件
```

### 模板化抽象工厂（编译时多态）

用 CRTP 和变参模板消除虚函数开销，适合性能敏感场景：

```cpp
// 编译时抽象工厂
template <typename ButtonType, typename MenuType>
class UIFactoryT {
public:
    static std::unique_ptr<Button> create_button() {
        return std::make_unique<ButtonType>();
    }
    static std::unique_ptr<Menu> create_menu() {
        return std::make_unique<MenuType>();
    }
};

// 定义具体工厂为类型别名
using WinUIFactory = UIFactoryT<WinButton, WinMenu>;
using MacUIFactory = UIFactoryT<MacButton, MacMenu>;

// 使用（完全编译时）
// auto btn = WinUIFactory::create_button();
```

## 优缺点

### ✅ 优点
- **产品族一致性**：同一工厂创建的产品天然兼容
- **隔离具体类**：客户端只依赖抽象接口
- **易于切换产品族**：替换工厂即可整体切换

### ❌ 缺点
- **扩展产品困难**：新增一个产品（如 `Dialog`）需要修改所有工厂接口和实现
- **类爆炸**：产品 × 平台的组合导致大量类
- **复杂度高**：简单场景杀鸡用牛刀

## 关键要点

> **抽象工厂解决的是"产品族"问题**——当你发现系统中有多组相关对象需要一起创建、一起切换时，就是抽象工厂的用武之地。

> **扩展产品 vs 扩展产品族**：抽象工厂对"新增产品族"友好（新增一组工厂），但对"新增产品类型"不友好（需改所有工厂接口）。这是它的核心权衡。

> **模板化变体**通过编译时绑定消除虚函数开销，但牺牲了运行时动态切换的能力。现代 C++ 中可根据场景选择。
