# Adapter 适配器模式

将已有类的接口转换为客户期望的接口，使原本因接口不兼容而无法协同工作的类能够一起工作。

## 意图与场景

当你有一个已有的类（`Adaptee`），其接口与客户端期望的接口不一致时，Adapter 充当转换层，而不修改已有代码。

**典型场景：**
- 封装第三方库，提供统一业务接口
- 集成遗留代码到新系统
- STL 容器适配器（`stack`、`queue`、`priority_queue` 就是底层容器的适配器）

**两种实现方式：**

| 方式 | 手段 | C++ 支持 |
|------|------|----------|
| 类适配器 | 多重继承 Target + Adaptee | 需要多重继承，耦合度高 |
| 对象适配器 | 组合（持有 Adaptee 指针） | 推荐，更灵活 |

## C++ 实现

### 对象适配器（推荐）

```cpp
#include <iostream>
#include <memory>
#include <string>

// 已有的不兼容接口
class XmlDataProvider {
public:
    std::string getXmlData() const {
        return "<data><item>42</item></data>";
    }
};

// 客户端期望的统一接口
class IDataProvider {
public:
    virtual ~IDataProvider() = default;
    virtual std::string getData() const = 0;
};

// 适配器：将 XmlDataProvider 适配到 IDataProvider
class XmlDataAdapter : public IDataProvider {
    std::unique_ptr<XmlDataProvider> adaptee_;
public:
    explicit XmlDataAdapter(std::unique_ptr<XmlDataProvider> adaptee)
        : adaptee_(std::move(adaptee)) {}

    std::string getData() const override {
        // 转换逻辑：从 XML 提取所需数据
        auto raw = adaptee_->getXmlData();
        // 简化示例：直接返回原始数据
        return "[JSON] " + raw;
    }
};

void processData(const IDataProvider& provider) {
    std::cout << provider.getData() << "\n";
}
```

### STL 容器适配器示例

`std::stack` 就是经典适配器——它以 `std::deque` 为底层容器，屏蔽掉不需要的接口：

```cpp
#include <stack>
#include <vector>
#include <deque>

int main() {
    // 默认用 deque 适配
    std::stack<int> s1;

    // 用 vector 适配
    std::stack<int, std::vector<int>> s2;

    s1.push(10);
    s1.push(20);
    // s1[0];  // 编译错误：stack 屏蔽了随机访问

    auto top = s1.top();  // 20
    s1.pop();
}
```

### 类适配器（多重继承）

```cpp
// 仅作展示，实际项目中慎用多重继承
class XmlClassAdapter : public IDataProvider, private XmlDataProvider {
public:
    std::string getData() const override {
        return getXmlData();  // 直接调用继承来的 adaptee 方法
    }
};
```

## 优缺点

**优点：**
- 单一职责：接口转换逻辑集中在一个类中
- 开闭原则：无需修改已有代码即可引入适配
- 可复用：同一个适配器可服务于多个客户端

**缺点：**
- 增加了一层间接调用
- 对象适配器对 Adaptee 有组合依赖
- 类适配器需要多重继承，C++ 中增加复杂度

## 关键要点

> - Adapter 解决的是「已有类接口不匹配」的问题，而非设计一个新接口
> - 对象适配器通过组合实现，比类适配器（多重继承）更灵活
> - STL 的 `stack`/`queue`/`priority_queue` 是 Adapter 模式的经典案例——它们适配底层容器的接口
> - 与 [[设计模式-外观模式|Facade]] 的区别：Facade 定义新接口简化子系统；Adapter 复用已有接口做转换
> - 与 [[设计模式-装饰器模式|Decorator]] 的区别：Adapter 改变接口；Decorator 增强功能但保持接口一致

## 相关模式
