内存屏障指令强制处理器的加载/存储操作按程序顺序执行，防止 CPU 和编译器的重排序优化破坏多线程同步语义。

## x86 内存序模型：TSO

x86/x64 使用 **TSO（Total Store Order）** 模型：

```
允许的重排序：
  ✅ Store → Load（存储后加载可以提前）
  ❌ Load → Load（不允许）
  ❌ Load → Store（不允许）
  ❌ Store → Store（不允许）
```

也就是说，x86 只允许一种重排序：**存储后加载**（Store-Load 可以乱序）。因此 x86 已经是"较强"的内存模型。

## 三条 Fence 指令

### SFENCE — 存储屏障

```nasm
; SFENCE：等待所有在 SFENCE 之前的存储操作完成
;         之后的存储不会重排到 SFENCE 之前

mov     [flag], 1              ; 写标志
sfence                          ; 确保 flag 写入完成
mov     [data], eax             ; 写数据
```

**用途**：结合非时间存储指令（MOVNTQ, MOVNTPS）使用。

### LFENCE — 加载屏障

```nasm
; LFENCE：等待所有在 LFENCE 之前的加载操作完成
;         之后的加载不会重排到 LFENCE 之前

mov     eax, [data]             ; 读数据
lfence                          ; 确保 data 读取完成
mov     ebx, [flag]             ; 读标志
```

**用途**：Spectre 缓解（序列化加载），防止推测执行。

### MFENCE — 完全屏障

```nasm
; MFENCE：等待所有在 MFENCE 之前的加载和存储完成
;         之后的任何访问不会重排到 MFENCE 之前

mov     [flag], 1               ; 写标志
mfence                          ; 完全屏障
mov     eax, [data]             ; 读数据 — 不会重排到写 flag 之前
```

**用途**：Store-Load 重排序的唯一屏障（x86 中其他重排序天然不发生）。

## 屏障指令对比

| 指令 | 限制的操作 | 速度 |
|------|-----------|------|
| `SFENCE` | Store → Store | 快 |
| `LFENCE` | Load → Load（且序列化） | 中 |
| `MFENCE` | 所有 Load/Store | 慢 |

## 编译器屏障 vs 硬件屏障

```c
// 编译器屏障 — 只阻止编译器重排，不阻止 CPU
__asm__ __volatile__ ("" ::: "memory");

// 硬件屏障 — 阻止 CPU 重排
__asm__ __volatile__ ("mfence" ::: "memory");

// 组合使用
__sync_synchronize();                // 编译器 + 硬件屏障
__atomic_thread_fence(__ATOMIC_SEQ_CST); // C11 内存屏障
```

### Linux 内核的宏

```c
// include/linux/compiler.h
#define barrier() __asm__ __volatile__ ("" ::: "memory")

// arch/x86/include/asm/barrier.h
#define mb()    __asm__ __volatile__ ("mfence" ::: "memory")
#define rmb()   __asm__ __volatile__ ("lfence" ::: "memory")
#define wmb()   __asm__ __volatile__ ("sfence" ::: "memory")
#define smp_mb()   barrier()    // 在 x86 上，编译器屏障就够了（TSO 保证）
#define smp_rmb()  barrier()
#define smp_wmb()  barrier()

// x86 上 Store-Load 需要真正的硬件屏障
#define smp_mb__after_atomic() mb()  // 原子操作后需要 MFENCE
```

## 实际应用：消息传递

```nasm
; 经典的消息传递模式
; 生产者：
    mov     [msg_data], eax          ; ① 写消息数据
    ; --- 需要 Store-Store 屏障 ---
    mfence                           ; 确保 ① 在 ② 之前完成
    mov     dword [msg_ready], 1     ; ② 设置 ready 标志

; 消费者：
.wait:
    mov     ebx, [msg_ready]
    test    ebx, ebx
    jz      .wait
    ; --- 此时 ready == 1，x86 TSO 保证 Load-Load 不重排 ---
    mov     ecx, [msg_data]          ; ③ 读消息数据（安全）
```

在 x86 上，由于 TSO 模型：
- ①→② 不会重排（Store-Store 天然有序）
- ③ 不会被提前到读 ready 之前（Load-Load 天然有序）
- 唯一需要屏障的是 Store-Load 模式

## LOCK 隐含 MFENCE

```nasm
; LOCK 前缀隐含完整的内存屏障
lock add    dword [counter], 1
; 等价于：
;   add [counter], 1
;   mfence

; 因此，原子操作后的内存访问不会重排到原子操作之前
```

## 关键要点

> **x86 是 TSO 模型**：只允许 Store→Load 重排序。其他重排序（Load→Load, Load→Store, Store→Store）天然不发生。这比 ARM/RISC-V 的弱序模型简单很多。

> **MFENCE 是最重的屏障**：只有 Store-Load 场景才需要。`SFENCE` 和 `LFENCE` 在 x86 上很少单独使用。

> **LOCK 隐含 MFENCE**：如果已经用了原子操作，额外的 MFENCE 是多余的。

> **编译器屏障 ≠ 硬件屏障**：`barrier()` 只阻止编译器重排，不阻止 CPU。但 x86 TSO 下，编译器屏障通常就够了（除了 Store-Load）。
