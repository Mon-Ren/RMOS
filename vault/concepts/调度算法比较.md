# 调度算法比较

## xv6：轮转调度（Round-Robin）

```
遍历 ptable → 找第一个 RUNNABLE → 运行 → yield → 下一个
```

- O(N) 扫描，无优先级，无时间片（靠时钟中断抢占）
- 简单、公平、可预测
- 不适合交互式和实时场景

## 经典算法对比

| 算法 | 策略 | 公平 | 响应 | 吞吐 | 复杂度 |
|------|------|------|------|------|--------|
| FCFS | 先来先服务 | ✗ | 差 | 一般 | O(1) |
| SJF | 最短作业优先 | ✗ | 一般 | 最优 | O(N) |
| 优先级 | 按优先级排序 | ✗ | 可调 | 可调 | O(logN) |
| **RR (xv6)** | 时间片轮转 | ✓ | 好 | 一般 | O(N) |
| MLFQ | 多级反馈队列 | ✗ | 优秀 | 好 | O(1)* |
| **CFS (Linux)** | 完全公平调度 | ✓ | 优秀 | 好 | O(logN) |

## Linux CFS（Completely Fair Scheduler）

核心思想：**每个进程获得等比例的 CPU 时间**。

### 虚拟运行时间（vruntime）

```c
vruntime += 实际运行时间 × (NICE_0_WEIGHT / 进程权重)
```

- 权重高的进程 vruntime 增长慢 → 更多运行机会
- 调度器总是选 vruntime 最小的进程
- 用红黑树维护 vruntime 有序 → O(logN) 插入/删除

### 权重与 nice 值

```
nice -20 (最高优先级) → 权重 88761
nice 0   (默认)       → 权重 1024
nice 19  (最低优先级) → 权重 15
```

权重比决定了 CPU 时间分配比例。

## FreeBSD / macOS：ULE 调度器

- 交互式优先：通过 CPU 使用量计算动态优先级
- 交互式进程（低 CPU 使用）优先级自动提升
- 后台计算进程优先级自动降低

## 实时调度

| 算法 | 特点 |
|------|------|
| Rate Monotonic | 静态优先级，周期短优先 |
| Earliest Deadline First | 动态优先级，截止期近优先 |

## 关键要点

> 从 xv6 的 RR 到 Linux 的 CFS，调度器的演化反映了对**公平性、响应性、吞吐量**三者平衡的追求。CFS 用 vruntime 统一了优先级和公平性——高优先级进程的 vruntime 增长慢，自然获得更多运行时间。红黑树让调度决策保持 O(logN)，是工程上的优雅选择。
