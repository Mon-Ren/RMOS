# 编译器优化与内联决策

编译器在 `-O2` 下自动做内联决策——理解它的启发式规则能帮你写出更易优化的代码。

## 编译器自动内联的条件

```
编译器倾向内联的场景：
- 函数体短小（通常 < 10 行，无复杂控制流）
- 调用点少（只调用一两次）
- 参数是常量表达式（常量传播后可能消除代码）
- 构造函数/析构函数中的简单操作
- Lambda 表达式（类型已知，天然可内联）

编译器拒绝内联的场景：
- 函数体大或复杂
- 递归（只内联有限层）
- 虚函数（除非编译器确定具体类型）
- 跨编译单元（除非 LTO）
- 函数地址被取（需要有实际地址）
```

## GCC/Clang 内联控制

```bash
# 查看内联决策报告
g++ -O2 -Winline main.cpp           # 对未内联的 inline 函数警告
g++ -O2 -fdump-ipa-inline main.cpp  # 输出内联决策日志

# 控制内联
__attribute__((always_inline)) 强制内联
__attribute__((noinline)) 禁止内联
__attribute__((flatten)) 内联所有被调用函数
```

## 帮助编译器内联

```cpp
// 1. 把热路径函数放在同一个编译单元
// 2. 用 LTO 跨编译单元内联
// 3. 用 PGO 让编译器知道哪些路径是热路径
// 4. 避免在热路径调用虚函数
// 5. 模板实例化在使用处（天然可内联）
```

## 关键要点

> 编译器的内联决策是启发式的——不同版本、不同优化级别结果不同。不要为了内联扭曲代码结构。

> LTO（链接时优化）是解决"跨编译单元无法内联"的标准方案——编译时 `-flto`，链接时也 `-flto`。
