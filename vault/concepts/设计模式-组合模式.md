# Composite 组合模式

将对象组合成树形结构以表示「部分-整体」层次结构，使客户端能以统一的方式处理单个对象和组合对象。

## 意图与场景

当业务模型具有递归的树形结构时，Composite 消除客户端对「叶子」和「容器」的区分，统一操作接口。

**典型场景：**
- 文件系统：File（叶子）+ Directory（可包含 File 和子 Directory）
- UI 组件树：Button（叶子）+ Panel（可包含子组件）
- 组织架构：Employee → Department（嵌套）
- AST（抽象语法树）：表达式的递归组合

**两种设计风格：**

| 风格 | 特点 | 风险 |
|------|------|------|
| 透明模式 | Component 基类声明所有方法（含 add/remove） | 叶子节点调用 add() 时运行时失败 |
| 安全模式 | 仅 Composite 声明 add/remove | 客户端需要区分叶子和组合类型 |

## C++ 实现

### 透明模式 — 文件系统

```cpp
#include <iostream>
#include <memory>
#include <string>
#include <vector>
#include <algorithm>
#include <numeric>
#include <format>

// --- Component 基类 ---
class FileSystemEntry {
protected:
    std::string name_;
public:
    explicit FileSystemEntry(std::string name) : name_(std::move(name)) {}
    virtual ~FileSystemEntry() = default;

    const std::string& name() const { return name_; }
    virtual size_t size() const = 0;
    virtual void print(std::ostream& os, int indent = 0) const = 0;

    // 透明模式：在基类声明，叶子提供默认实现
    virtual void add(std::unique_ptr<FileSystemEntry>) {
        throw std::logic_error{name_ + " is not a directory"};
    }
    virtual void remove(const std::string&) {
        throw std::logic_error{name_ + " is not a directory"};
    }
};

// --- Leaf ---
class File : public FileSystemEntry {
    size_t size_;
public:
    File(std::string name, size_t size)
        : FileSystemEntry(std::move(name)), size_(size) {}

    size_t size() const override { return size_; }

    void print(std::ostream& os, int indent = 0) const override {
        os << std::string(indent, ' ') << "📄 " << name_
           << " (" << size_ << " bytes)\n";
    }
};

// --- Composite ---
class Directory : public FileSystemEntry {
    std::vector<std::unique_ptr<FileSystemEntry>> children_;
public:
    explicit Directory(std::string name)
        : FileSystemEntry(std::move(name)) {}

    size_t size() const override {
        return std::accumulate(children_.begin(), children_.end(), size_t{0},
            [](size_t sum, const auto& child) { return sum + child->size(); });
    }

    void add(std::unique_ptr<FileSystemEntry> entry) override {
        children_.push_back(std::move(entry));
    }

    void remove(const std::string& entryName) override {
        std::erase_if(children_, [&](const auto& e) {
            return e->name() == entryName;
        });
    }

    void print(std::ostream& os, int indent = 0) const override {
        os << std::string(indent, ' ') << "📁 " << name_ << "/\n";
        for (const auto& child : children_) {
            child->print(os, indent + 2);
        }
    }
};

// --- 使用：客户端无需区分 File 和 Directory ---
int main() {
    auto root = std::make_unique<Directory>("project");
    root->add(std::make_unique<File>("CMakeLists.txt", 256));

    auto src = std::make_unique<Directory>("src");
    src->add(std::make_unique<File>("main.cpp", 4096));
    src->add(std::make_unique<File>("utils.cpp", 2048));

    root->add(std::move(src));

    // 统一接口
    std::cout << "Total size: " << root->size() << " bytes\n\n";
    root->print(std::cout);
}
```

输出：
```
Total size: 6400 bytes

📁 project/
  📄 CMakeLists.txt (256 bytes)
  📄 src/
    📄 main.cpp (4096 bytes)
    📄 utils.cpp (2048 bytes)
```

### 安全模式

```cpp
// 叶子只实现叶子相关的接口
class Leaf : public Component {
    // 没有 add/remove 方法
};

// 组合类才声明 add/remove
class Composite : public Component {
public:
    void add(std::unique_ptr<Component> child);
    void remove(const std::string& name);
    // 客户端需要 dynamic_cast 或知道具体类型
};
```

## 优缺点

**优点：**
- 客户端代码一致：处理叶子和组合对象的代码相同
- 符合开闭原则：新增组件类型不影响现有代码
- 自然表达递归树形结构

**缺点：**
- 限制子元素类型困难（透明模式下所有 Component 类型都能 add）
- 叶子节点可能被迫实现不相关的接口（透明模式）
- 树很深时，调试和遍历开销增大

## 关键要点

> - Composite 模式的核心是「递归组合 + 统一接口」——让叶子和容器共享同一抽象
> - 透明模式让客户端更简单，但叶子节点可能有无意义的 add/remove 方法
> - 安全模式更类型安全，但客户端需要区分叶子和组合
> - `std::filesystem::path` 的 `/` 操作符也体现了类似的组合思想
> - 递归遍历（size、print）是 Composite 的经典操作模式
