# KB Expander

系统性地扩充双链笔记知识库。不是随机写，而是有策略地覆盖知识体系。

## 工作流程

### Phase 1: 知识图谱审计

先了解现状，再决定写什么。

```bash
# 1. 查看当前 MOC
cat vault/MOC.md

# 2. 查看链接关系，找孤立节点
python3 scripts/obsidian_kb.py -v vault graph

# 3. 查找引用了但不存在的笔记（死链接）
python3 scripts/kb_audit.py -v vault
```

### Phase 2: 确定扩展方向

根据审计结果，从三个维度扩展：

| 策略 | 说明 | 优先级 |
|------|------|--------|
| **深度挖掘** | 已有笔记的子主题，展开细节 | 高 |
| **横向关联** | 相邻领域的交叉主题 | 中 |
| **前沿延伸** | 现代 OS 的演化方向 | 低 |

每篇新笔记必须关联至少 2 篇已有笔记。

### Phase 3: 内容生产

#### 来源优先级

1. **源码** — 直接读代码，最准确
2. **官方文档** — 架构设计、接口说明
3. **学术论文** — 原始设计动机
4. **网络搜索** — 补充理解、找到示例

#### 生成流程

对每个新主题：

```
1. 确定主题和类型（concept/paper/code/project）
2. 搜索相关资料（源码 + 网络）
3. 用 new 创建笔记
4. 填充内容：
   - 定义/概述（是什么）
   - 原理/机制（怎么工作）
   - 代码示例（xv6/其他 OS 实现）
   - 关键要点（一句话核心）
5. 建立双向链接（至少 2 个）
6. 回链：更新已有笔记，加入指向新笔记的链接
```

### Phase 4: 质量检查

```bash
# 更新 MOC
python3 scripts/obsidian_kb.py -v vault moc

# 检查链接完整性
python3 scripts/obsidian_kb.py -v vault graph

# 搜索测试：验证检索效果
python3 scripts/obsidian_kb.py -v vault ask "测试问题"
```

## 笔记写作规范

### 原子性

一篇笔记 = 一个概念。不要把"页表机制"和"虚拟内存"混在一篇。

### 结构模板

```markdown
# 标题

一句话概述。

## 核心机制
（技术细节，配代码/图示）

## 与 X 的对比/关系
（横向联系）

## 关键要点
> 一句话核心结论

## 关联
- [[已有笔记1]] — 关系说明
- [[已有笔记2]] — 关系说明
```

### 代码注释

xv6 源码引用时，标注文件名和关键行：

```c
// proc.c:59 — allocproc 分配进程槽
static struct proc* allocproc(void) {
```

### 避免的模式

- ❌ 大段抄书/抄文档（用自己的话总结）
- ❌ 没有代码支撑的纯理论
- ❌ 孤立笔记（不链接其他笔记）
- ❌ 标题太宽泛（"操作系统" → 应该拆分）

## 批量扩展策略

### 单次会话扩展

一次扩展 3-5 篇相关主题，形成小集群：

```
例：围绕「xv6 文件系统」扩展
  → buffer cache（已有 fs.c）
  → 日志系统原理（已有日志层）
  → inode 详解（已有 inode 缓存）
  → 目录与路径解析（已有名称层）
  → 磁盘调度算法（新）
```

### 交叉引用检查

写完一批后，检查：
- 每篇新笔记是否有 `## 关联` 指向已有笔记？
- 已有笔记是否回链了新笔记？
- MOC 是否包含所有新笔记？

## 对话式使用示例

```
用户: 帮我围绕「xv6 进程管理」扩展相关知识

Agent:
1. 查看 vault/MOC.md 和 graph，了解当前覆盖
2. 发现缺失：信号处理、exec 详解、僵尸进程
3. 搜索 xv6 源码中 exec.c、signal 相关代码
4. 逐篇创建笔记，建立链接
5. 更新已有笔记的关联部分
6. 更新 MOC
```
